# AGENTS.md

Guidance for any agent or contributor working in this repository.

## Project

Console maze program written in **C**. It generates mazes, reads/writes them to
files, lets a player walk a maze manually, and lets the computer solve a maze
automatically with visualised step-by-step traversal. Driven by a console menu.

This is a 3-person school project (Технологично училище "Електронни системи",
ТУ-София — лятна практика, 10 клас). Work is split by module (see Ownership).

## Hard rules

- **Language: C only** (C99). No C++, no other languages.
- **No comments anywhere.** Not in `.c`, not in `.h`. Code must read clearly on
  its own through descriptive names and small functions. This is a strict
  requirement — do not add explanatory comments, banners, or TODO notes.
- **No co-author / tool attribution in commits.** Do not add `Co-Authored-By`
  trailers or any "generated by" lines to commit messages.
- Target platform is **macOS / Linux**. Use `usleep` (from `<unistd.h>`) for
  delays and ANSI escape codes for clearing the screen / cursor moves. Do not
  pull in `<windows.h>` or `system("cls")`.

## Build & run

```bash
make          # builds ./maze
make clean
./maze        # launches the menu
```

Compiler flags: `-std=c99 -Wall -Wextra -O2`. Code must build warning-free.

## Shared data contract (all modules depend on this)

Defined in `maze.h`. Do not change without coordinating across modules.

```c
#define MAZE_MIN     3
#define MAZE_MAX_W   40
#define MAZE_MAX_H   30
#define WALL         '#'
#define EMPTY        ' '

typedef struct {
    int  width;
    int  height;
    char cells[MAZE_MAX_H][MAZE_MAX_W];
} Maze;
```

- Indexing is `cells[row][col]`, `row` in `[0, height)`, `col in [0, width)`.
- Start is the top-left cell `(0, 0)`. Goal is the bottom-right cell
  `(height-1, width-1)`. Both are always `EMPTY` and always connected by at
  least one path.
- Size limits: width `3..40`, height `3..30`.

### File format

Plain text. First line is `width height` (two integers, space separated).
Then `height` lines, each exactly `width` characters, using `#` for wall and a
space for empty — i.e. the maze drawn exactly as it appears on screen.

```
10 6
##########
#        #
# ## ### #
# #    # #
#   ##   #
##########
```

A reader must validate: dimensions within limits, correct line count, correct
line lengths, and only legal characters.

## Module ownership

| Area                              | File(s)              | Owner       |
|-----------------------------------|----------------------|-------------|
| Shared types / drawing helper     | `maze.h`, `maze.c`   | shared      |
| Maze generation + seeded RNG      | `generate.c`         | Georgi      |
| File read/write                   | `fileio.c`           | teammate 2  |
| Player (manual) traversal         | `traverse.c`         | teammate 2  |
| Computer solving (2+ algorithms)  | `solve.c`            | teammate 3  |
| Menu / program entry              | `main.c`             | teammate 3  |

Each module exposes its functions through `maze.h` (or a small per-module
header) and keeps internal helpers `static`.

## Commit conventions

- Small, focused commits with imperative subject lines
  (e.g. `Add recursive backtracker carving`).
- One logical change per commit. Aim for frequent commits.
- Author attribution is assigned per module owner; commit dates are normalised
  at the end of the project. Build features in their natural order so the
  history reads sensibly once dates are applied.
- No tool/co-author trailers (see Hard rules).
